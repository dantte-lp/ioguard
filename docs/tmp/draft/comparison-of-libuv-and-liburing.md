# libuv vs liburing: выбор фундамента для VPN-сервера нового поколения

**io_uring превосходит libuv по «сырой» производительности сетевого I/O на 25–43%, но для проекта ocserv-modern оптимальна гибридная архитектура** — io_uring на горячем пути данных и libuv (или минимальный слой управления) для контрольной плоскости. Причина: io_uring даёт кратное сокращение системных вызовов и поддержку zero-copy, критичную для VPN-туннелей с тысячами одновременных сессий, но не имеет встроенных механизмов DNS-резолвинга, управления процессами и кроссплатформенных абстракций таймеров/сигналов. При этом ваш внутренний документ «Рефакторинг Ocserv» рекомендует миграцию с libev на libuv — эта рекомендация остаётся разумной отправной точкой, с последующей оптимизацией горячего пути через io_uring.

---

## 1. Архитектура: реактивность против проактивности

Фундаментальное различие между libuv и io_uring — это разница между **readiness-based** (готовность) и **completion-based** (завершение) моделями I/O.

**libuv** реализует классический событийный цикл с обратными вызовами. На Linux он использует epoll как основной бэкенд для сетевого I/O. Цикл проходит **9 фаз** за каждую итерацию: обновление временной метки → выполнение таймеров → обработка отложенных callback'ов → idle/prepare хэндлы → **блокирующий epoll_wait** → check хэндлы → close callback'и. Абстракция построена на двух примитивах: **хэндлы** (долгоживущие объекты — `uv_tcp_t`, `uv_udp_t`, `uv_timer_t`, `uv_signal_t`) и **запросы** (кратковременные операции — `uv_write_t`, `uv_connect_t`). Файловый I/O выполняется в **пуле потоков** (по умолчанию 4, до 1024 через `UV_THREADPOOL_SIZE`), поскольку большинство файловых систем не поддерживают настоящий асинхронный I/O. С версии **v1.45.0** (июнь 2023) файловые операции могут использовать io_uring внутри, что дало **8-кратный рост пропускной способности** файлового I/O. С версии **v1.50.0** (январь 2025) io_uring используется также для батчинга epoll-операций.

**io_uring** — принципиально иная модель: два кольцевых буфера в разделяемой памяти между userspace и ядром. **Submission Queue (SQ)** — приложение добавляет запросы (SQE по 64 байта), ядро потребляет. **Completion Queue (CQ)** — ядро записывает результаты (CQE по 16 байт), приложение читает. Обе очереди реализованы как lockless single-producer/single-consumer кольца степени двойки. Это означает: ядро само выполняет операцию и возвращает готовый результат, а не уведомляет о готовности. Три режима работы: **interrupt-driven** (стандартный, с `io_uring_enter()`), **SQPOLL** (kernel-поток непрерывно опрашивает SQ — **ноль системных вызовов** на горячем пути) и **IOPOLL** (busy-polling для NVMe/O_DIRECT).

| Характеристика      | libuv (epoll)                                                                   | io_uring (liburing)                                  |
| ------------------- | ------------------------------------------------------------------------------- | ---------------------------------------------------- |
| Модель уведомления  | «FD готов к чтению/записи» → нужен отдельный syscall                            | «Чтение/запись завершено» → данные уже готовы        |
| Syscall'ы на I/O    | Минимум 2: epoll_wait + read/write                                              | Амортизируется до 0 с SQPOLL                         |
| Батчинг             | Ограничен: epoll_wait возвращает события, каждый read/write — отдельный syscall | Нативный: N операций за один io_uring_enter          |
| Файловый I/O        | Пул потоков (блокирующий)                                                       | Истинный async в ядре                                |
| Управление буферами | Callback-аллокатор, пользователь управляет памятью                              | Registered buffers, provided buffer rings, zero-copy |
| Унификация          | Разные пути для сети (epoll) и файлов (threadpool)                              | Единый интерфейс для всех типов I/O                  |

## 2. Бенчмарки: когда io_uring выигрывает и когда нет

Результаты бенчмарков зависят от уровня конкурентности, размера батчей и характера нагрузки. **io_uring — не серебряная пуля**: наивная замена epoll даёт лишь ~6% прироста, но оптимизированная реализация может дать двукратное и более преимущество.

**Сокращение системных вызовов** — главное преимущество io_uring для VPN. Red Hat (апрель 2023) продемонстрировал echo-сервер: 50 одновременных клиентов, 1000 запросов каждый — **5000 сокетных операций за 23 syscall'а** через io_uring против ~15 000+ syscall'ов через epoll. Это **сокращение в 650 раз**. При проектировании VPN-сервера, обрабатывающего тысячи пакетов в секунду через TUN-устройство, такая разница непосредственно транслируется в снижение нагрузки на CPU.

**Пропускная способность и задержки** по данным нескольких независимых тестов:

| Сценарий                                             | epoll       | io_uring        | Дельта           | Источник            |
| ---------------------------------------------------- | ----------- | --------------- | ---------------- | ------------------- |
| Echo-сервер, 40–400 соединений                       | Быстрее     | Медленнее       | epoll выигрывает | uWebSockets, 2020   |
| Echo-сервер, 1000 соединений (с Spectre mitigations) | Базовый     | +10% throughput | io_uring         | Alibaba, 2022       |
| TCP echo, p99 latency                                | Выше        | ~1мс ниже       | +25% throughput  | Ryan Seipp          |
| Job queue push latency, P99, 100K/s                  | 1.8мс       | 0.9мс           | −50%             | flashQ              |
| Сетевой throughput, ZC send, MTU 1500                | 74.4 Gbps   | 106.7 Gbps      | **+43%**         | Kernel Recipes 2024 |
| Сетевой throughput, ZC send, MTU 4096                | 82.2 Gbps   | 116.2 Gbps      | **+41%**         | Kernel Recipes 2024 |
| UDP ZC send, 65K payload                             | 47 421 MB/s | 116 079 MB/s    | **×2.44**        | Ядро Linux, патчи   |
| DBMS network shuffle, оптимизированный + ZC Rx       | Базовый     | до ×2.5         | Существенно      | arXiv, дек. 2025    |
| DBMS, наивная замена                                 | Базовый     | ×1.06           | Маргинально      | arXiv, дек. 2025    |

Анализ Alibaba (ноябрь 2022) количественно объяснил, _почему_ io_uring выигрывает не всегда. Накладные расходы на стороне ядра: epoll send/recv — **~1500 нс**, фреймворк io_uring — **~1900 нс** (на 27% больше). Стоимость переключения контекста syscall — **~700 нс** с включёнными защитами Spectre/Meltdown. При батче из N операций io_uring амортизирует эти 700 нс, и при **N > ~3** начинает выигрывать. Защиты от Spectre _усиливают_ преимущество io_uring: чем дороже syscall, тем выгоднее батчинг.

**Zero-copy** — это область, где io_uring демонстрирует максимальное преимущество, критичное для VPN. `IORING_OP_SEND_ZC` (ядро 6.0+) позволяет отправлять данные без копирования в буфер ядра. Для UDP-пакетов размером 65 КБ io_uring ZC показал **2.44× производительности** по сравнению с MSG_ZEROCOPY. Для VPN-сервера, пересылающего зашифрованные пакеты между TUN-устройством и сетевым сокетом, zero-copy может значительно снизить потребление памяти и CPU.

**Envoy Proxy** (экспериментальная интеграция, GitHub issue #28395) показал на реальном reverse-proxy: при 128 соединениях io_uring дал **+18% RPS** и **−14% latency** по сравнению с epoll-бэкендом — в одно-поточном режиме.

## 3. Сложность разработки и интеграция с TLS

Разница в сложности API между libuv и liburing существенна — это не просто вопрос синтаксиса, а разные ментальные модели программирования.

**libuv** предоставляет привычную callback-модель. TCP-сервер требует ~50–80 строк C-кода: `uv_tcp_init` → `uv_tcp_bind` → `uv_listen` → в callback'е `uv_accept` → `uv_read_start`. Управление ошибками — через отрицательные коды возврата (`UV_ECONNRESET`), единообразные по всему API. Буферы выделяются через пользовательский `alloc_cb`, вызываемый перед каждым чтением. Модель потоков строго определена: **один цикл на поток**, межпоточная коммуникация через `uv_async_send()` — единственный потокобезопасный примитив. Встроенный пул потоков для блокирующих операций (PAM-аутентификация, DNS).

**liburing** требует явного управления конечным автоматом. TCP-сервер — ~100–150 строк C. Разработчик оперирует SQE/CQE напрямую: `io_uring_get_sqe` → `io_uring_prep_accept` → `io_uring_sqe_set_data` (тегирование контекстом) → `io_uring_submit` → `io_uring_wait_cqe` → диспатч по типу события через switch/case. Ошибки — через `cqe->res` (отрицательный errno), что требует знания семантики нижележащих системных вызовов. Буферы **должны оставаться валидными** до появления CQE — преждевременное освобождение ведёт к UB. Продвинутые паттерны: **registered buffers** (предварительная фиксация страниц в ядре), **provided buffer rings** (ядро само выбирает буфер из пула), **multishot accept/recv** (один SQE генерирует множество CQE).

### Интеграция с TLS — критический аспект для VPN

Для libuv стандартный подход — **BIO pairs** OpenSSL/wolfSSL: создание пары BIO-объектов, привязка к SSL-сессии, ручная перекачка данных между libuv-потоком и SSL-стеком. Это рабочая, хорошо документированная схема, но она добавляет значительную сложность из-за необходимости обрабатывать renegotiation (когда `SSL_read` требует записи и наоборот). Зрелых библиотек-обёрток (libuv-tls, evt-tls) не существует в production-качестве.

Для io_uring два пути. **Первый** — userspace TLS, аналогичный подходу libuv: io_uring читает зашифрованные данные → передача через SSL_read в userspace → обработка. **Второй** — **kTLS** (kernel TLS, Linux 4.13+): после TLS-хэндшейка в userspace ключи передаются в ядро через `setsockopt(SOL_TLS)`, и все последующие send/recv автоматически шифруются/дешифруются ядром. Бенчмарки показывают **8–29% ускорения** на nginx и до **58% ускорения** файловых трансферов по сравнению с userspace-шифрованием. Комбинация io_uring + kTLS позволяет построить **HTTPS-сервер с нулём syscall'ов** в steady state (продемонстрировано проектом tarweb).

**Критическое ограничение для VPN: kTLS не поддерживает DTLS.** Основной канал данных OpenConnect VPN — UDP/DTLS. Это означает, что для горячего пути данных (DTLS-пакеты) кTLS неприменим, и шифрование в любом случае происходит в userspace через wolfSSL/OpenSSL. kTLS полезен только для TCP/TLS-канала управления и резервного канала данных. Это существенно уменьшает преимущество io_uring + kTLS в конкретном VPN-сценарии.

### Модели многопоточности

libuv использует модель **один цикл на поток** с `uv_async_send()` для межпоточного взаимодействия. io_uring рекомендует **одно кольцо на поток** (привязка к ядру CPU), с `IORING_SETUP_SINGLE_ISSUER` для оптимизации. Режим SQPOLL создаёт отдельный kernel-поток для опроса SQ, что **полностью исключает syscall'ы** для submit'а на загруженном сервере, но требует `CAP_SYS_ADMIN` или kernel ≥ 5.13. `IORING_SETUP_ATTACH_WQ` позволяет нескольким кольцам разделять async-воркеры, уменьшая число kernel-потоков.

## 4. VPN-специфика: TUN-устройства, таймеры, сигналы

Ключевые аспекты для ocserv-modern заслуживают отдельного анализа, поскольку здесь решения libuv и io_uring имеют критические различия в зрелости.

**Одновременная работа с TCP и UDP** — базовая функция обеих библиотек. libuv: `uv_tcp_t` + `uv_udp_t` в одном event loop. io_uring: `IORING_OP_RECV`/`IORING_OP_SEND` для обоих типов сокетов в одном кольце. Здесь паритет.

**TUN-устройства** — область, где io_uring пока имеет проблемы. Документированные в liburing issue #239 сложности включают: ошибки `-EINVAL` на ядрах старше 5.6, необходимость non-blocking режима (вызывающая постоянные `EAGAIN`), зависания записи, и **полная неработоспособность SQPOLL с TUN** (ошибка «bad file descriptor»). Мотивация для использования io_uring с TUN очевидна: «с mitigations=auto производительность TUN через read/write снижается на 30–40% при высоких полосах пропускания» из-за одного syscall'а на пакет при 40–50 пакетах за итерацию цикла. Батчинг io_uring мог бы решить эту проблему, но совместимость драйвера TUN пока неидеальна. libuv обёртывает TUN fd через `uv_poll_t` — стабильно работающий, проверенный подход.

**Таймеры**: libuv предоставляет `uv_timer_t` (min-heap внутри, проверка в начале каждой итерации). io_uring — `IORING_OP_TIMEOUT` (SQE в том же кольце, CQE по истечении). Преимущество io_uring: таймеры, I/O и всё остальное — в единой очереди завершений. Для VPN-сервера это означает единый цикл обработки keepalive'ов, session timeout'ов и DPD вместе с сетевыми событиями.

**Сигналы**: libuv имеет `uv_signal_t` — кроссплатформенная обработка SIGHUP/SIGTERM, интегрированная в цикл. io_uring потребует связки signalfd + `IORING_OP_READ` — работоспособно, но менее элегантно.

**DNS**: libuv имеет встроенный `uv_getaddrinfo()` (выполняется в пуле потоков). io_uring **не имеет** нативной поддержки DNS — потребуется внешний резолвер (c-ares) или блокирующий вызов в io_uring worker.

**Fork/process management**: libuv предоставляет `uv_spawn()`. Кольца io_uring **не наследуются через fork()** — дочерние процессы должны создавать собственные. Для ocserv, использующего connect-script/disconnect-script, это не критично, но требует внимания при архитектуре worker-процессов.

## 5. Как реальные проекты решают эту задачу

Ни один крупный VPN-проект не использует io_uring в production на момент марта 2026, но тенденция к адаптации ускоряется в прокси-серверах.

**WireGuard** использует **синхронную** криптографию в модуле ядра (~4000 строк C) — работает напрямую через networking stack ядра, без event loop. Userspace-реализации: wireguard-go (горутины Go), BoringTun от Cloudflare (async Rust). WireGuard работает только по UDP, что значительно проще двухканальной модели OpenConnect.

**OpenVPN** использует **собственный event loop** на базе `select()`/`poll()` — технологию, предшествующую и libuv, и io_uring. Его внутренний цикл мониторит сетевой сокет, TUN-интерфейс и расписание событий. Модернизации на io_uring или libuv не проводилось.

**ocserv** (текущий) использует **libev** (Reactor pattern) с моделью «один процесс на пользователя», привилегированный master-процесс + security-модуль + sandboxed worker'ы через seccomp. Ваш внутренний документ «Рефакторинг Ocserv» рекомендует миграцию на libuv с переходом от процессной модели к пулу потоков фиксированного размера.

**Cloudflare** активно использует io_uring в инфраструктуре, подробно документируя паттерны (NUMA-aware worker pools, bounded/unbounded воркеры). Их proxy-фреймворк **Oxy** и reverse-proxy **Pingora** написаны на Rust с async runtime. **BoringTun** — их production userspace WireGuard — развёрнут на «миллионах устройств iOS/Android и тысячах Linux-серверов».

**Envoy** имеет экспериментальную поддержку io_uring (третья итерация, issue #28395): при 128 соединениях — **+18% RPS, −14% latency**. Ещё не production-ready.

**nginx** получил патч от Intel (февраль 2021) для замены Linux AIO на io_uring: **3× ускорение** для 4КБ файлов (220 МБ/с vs 70 МБ/с в single-worker режиме). Экспериментальный форк nginx-io_uring заменяет epoll на io_uring целиком.

**HAProxy** использует io_uring для батчевой загрузки SSL-сертификатов: **25% ускорение** при загрузке 24 686 сертификатов с пустым кэшем.

**cjdns** — «encrypted self-configuring network/VPN routing engine» — **использует libuv** и является наиболее релевантным VPN-проектом на libuv.

## 6. Платформенная поддержка и безопасность

Для Linux-only VPN-сервера кроссплатформенность libuv — избыточное преимущество, но экосистема безопасности различается кардинально.

**libuv** поддерживает Linux (≥ 2.6.32), macOS (≥ 10.7), Windows (≥ 8), FreeBSD (≥ 10), AIX, z/OS, Solaris. Для ocserv-modern, нацеленного исключительно на Linux, эта широта не нужна, но libuv имеет **безупречный послужной список безопасности**: всего 4 CVE за 14 лет (все — userspace-баги, быстро исправленные).

**io_uring** — исключительно Linux. Минимальные версии ядра для ключевых функций:

| Функция                                   | Ядро | Значение для VPN     |
| ----------------------------------------- | ---- | -------------------- |
| Базовый io_uring                          | 5.1  | Минимум              |
| FAST_POLL (без async-потоков для сокетов) | 5.7  | Критично             |
| SQPOLL без root                           | 5.13 | Важно для production |
| Multishot accept                          | 5.19 | Полезно              |
| Multishot recv, SEND_ZC                   | 6.0  | Очень полезно        |
| SENDMSG_ZC                                | 6.1  | Полезно для UDP      |
| Recv/send bundles                         | 6.10 | Оптимизация          |

**Безопасность io_uring — серьёзная тема.** Google сообщил (июнь 2023), что **60% kernel-эксплойтов** в их программе bug bounty kCTF в 2022 году использовали io_uring; Google выплатил ~$1 млн за уязвимости. Результат: io_uring **отключён** в Android, ChromeOS и на production-серверах Google. ARMO продемонстрировал (апрель 2025) rootkit «Curing», обходящий Falco, Microsoft Defender, CrowdStrike через io_uring (операции не триггерят syscall-хуки). Docker блокирует `io_uring_setup` по умолчанию через seccomp-профиль.

**Для VPN-сервера как системного сервиса** эти ограничения менее критичны: сервер — доверенный компонент (consistent с рекомендацией Google: «safe for use by trusted components»), работает вне контейнера, уязвимости ядра закрываются обновлениями. Но это означает повышенные требования к версии ядра и оперативным обновлениям.

## 7. Гибридные подходы и эволюция libuv

**libuv уже использует io_uring** как бэкенд — но пока ограниченно. С v1.45.0 — файловые операции (read, write, fsync, stat). С v1.50.0 — батчинг epoll-операций. **Сетевой I/O по-прежнему через epoll.** В issue #4044 обсуждается перенос сетевого I/O на io_uring, но это требует фундаментального изменения API (с readiness-based на completion-based), что означает потенциальный libuv v2.

Среди других event loop библиотек: **libev** имеет io_uring-бэкенд (`ev_iouring.c`) для опроса fd с v4.33. **libevent** — нет поддержки (открытый issue с 2019). **libxev** (Zig) — полноценный io_uring-бэкенд с proactor API.

**Гибридная архитектура data path / control plane** — практически осуществимый подход:

- **io_uring** для горячего пути данных: чтение из TUN → шифрование → отправка в сокет (и обратная цепочка), с батчингом и registered buffers
- **libuv** (или минимальный цикл на epoll) для контрольной плоскости: таймеры, сигналы, DNS, process management, конфигурация
- Связь между двумя через **eventfd**: зарегистрировать eventfd в io_uring, добавить его в epoll-цикл libuv

Этот подход используется неявно самим libuv (файлы через io_uring, сеть через epoll) и может быть реализован явно на уровне архитектуры ocserv-modern.

## Рекомендация для ocserv-modern

Оптимальная стратегия для Linux-only VPN-сервера на C23 — **фазовый подход** с гибридной архитектурой.

**Фаза 1**: миграция с libev на libuv как основной event loop (совпадает с рекомендацией вашего внутреннего документа). Это даёт: стабильный API, встроенный пул потоков для аутентификации и DNS, зрелую обработку сигналов и процессов, io_uring для файловых операций «бесплатно». Переход от процессной модели к пулу потоков. Этого достаточно для **первого production-релиза**.

**Фаза 2**: профилирование горячего пути данных (TUN ↔ socket forwarding). Если bottleneck — syscall overhead (что вероятно при >1000 одновременных туннелей), внедрение io_uring через liburing для data plane: батчинг read/write на TUN и сетевых сокетах, registered buffers для буферного пула пакетов, `IORING_OP_SEND_ZC` для отправки зашифрованных пакетов. TLS/DTLS остаётся в userspace (wolfSSL) из-за отсутствия поддержки DTLS в kTLS. Контрольная плоскость остаётся на libuv.

**Фаза 3** (опционально, при нацеленности на максимальную производительность): полная миграция на io_uring с минимальной обёрткой для таймеров, сигналов и DNS. kTLS для TCP/TLS-канала управления. SQPOLL для серверов с высокой нагрузкой. Требует kernel ≥ 6.0 для полного набора функций.

Такой подход минимизирует риск (libuv — проверенная технология) при сохранении пути к максимальной производительности через io_uring, и позволяет принимать решения о каждой фазе на основе реальных данных профилирования, а не теоретических бенчмарков.
