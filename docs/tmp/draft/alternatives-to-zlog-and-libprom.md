# Альтернативы zlog и libprom для современных проектов на C

**Экосистема чистых C-библиотек для логирования и мониторинга Prometheus фрагментирована, но содержит несколько достойных вариантов.** Для логирования наиболее перспективная замена zlog — **stumpless** (структурированное логирование по RFC 5424, активная разработка, кроссплатформенность). Для Prometheus-метрик — **jelmd/libprom**, наиболее активно поддерживаемый форк оригинальной digitalocean/prometheus-client-c, уже включаемый в Debian. Ни одна из обнаруженных чисто C-библиотек не заявляет явную поддержку C17/C23, большинство ориентированы на C99/C11, однако они компилируются современными компиляторами без проблем.

## Часть I. Альтернативы zlog — библиотеки логирования на C

### Текущее состояние zlog

Библиотека **zlog** (HardySimpson/zlog) остаётся одним из наиболее полнофункциональных решений для логирования на чистом C. Репозиторий на GitHub набрал **~2 500 звёзд** и 765 форков при 403 коммитах. Лицензия — **LGPL-2.1** (важный момент: при статической линковке необходимо соблюдать условия LGPL). Библиотека управляется через конфигурационный файл с моделью «категории → форматы → правила», поддерживает горячую перезагрузку конфигурации, ротацию файлов по размеру, вывод в файлы, stdout/stderr, syslog и пользовательские обработчики. Производительность — около **250 000 записей/сек**, что примерно в 1000 раз быстрее syslog(3) с rsyslogd. Потокобезопасность обеспечена через pthreads. Основные ограничения: работает **только на POSIX-системах** (нет нативной поддержки Windows), требует сборки как разделяемая библиотека, конфигурация только через файл, проект находится в режиме поддержки без активного развития.

Существуют форки: **lisongmin/zlog** (CMake-версия), **lopsd07/WinZlog** (порт на Windows), **bmanojlovic/zlog** (autotools-версия). Также есть одноимённая, но совершенно другая библиотека **zma/zlog** — простой буферный логгер в памяти (public domain).

### stumpless — наиболее функциональная альтернатива

| Параметр | Значение |
|----------|----------|
| GitHub | https://github.com/goatshriek/stumpless |
| Звёзды | **~486** |
| Лицензия | Apache-2.0 |
| Язык | Чистый C |
| Сборка | CMake |
| Зависимости | Нет внешних (только CMake) |
| Статус | **Активно развивается** |

**stumpless** — наиболее серьёзный кандидат на замену zlog для проектов, которым нужна полнофункциональная библиотека логирования на чистом C. Ключевое преимущество — **структурированное логирование по RFC 5424**, чего нет у zlog. Библиотека поддерживает множество целей вывода: файлы, сеть (UDP4/TCP4), syslog (/dev/log), journald, Windows Event Log, **Splunk**, rsyslog, **SQLite** и пользовательские функции. Работает кроссплатформенно: Linux, Windows, macOS, Cygwin, FreeBSD, DOS.

API предоставляет как полный интерфейс управления целями вывода, так и удобные сокращения:

```c
stumplog(LOG_INFO | LOG_USER, "Message #%d", count);  // drop-in замена syslog()
stump_i("Информационное сообщение");                    // сокращённый вызов
stump_i_t("С информацией о месте вызова");              // автоматический файл:строка
```

Макросы `STUMPLESS_ENABLE_UPTO` / `STUMPLESS_DISABLE_DOWNTO` позволяют полностью исключить неиспользуемые уровни на этапе компиляции. Документация генерируется через Doxygen, есть man-страницы и примеры. Библиотека **интернационализирована** (12+ языков). Потокобезопасна.

**Минусы по сравнению с zlog**: нет системы конфигурационных файлов (только программная настройка), нет встроенной ротации логов, значительно меньше сообщество.

### pymumu/tinylog — асинхронное логирование для Linux

| Параметр | Значение |
|----------|----------|
| GitHub | https://github.com/pymumu/tinylog |
| Лицензия | MIT |
| Язык | C (C99+) |
| Сборка | Makefile |
| Зависимости | pthreads, zlib |
| Платформа | **Только Linux/UNIX** |

Уникальная черта tinylog — **потоко- и процессобезопасность**: несколько процессов могут записывать в один лог-файл. Логирование **асинхронное** по умолчанию, поддерживается неблокирующий режим (`TLOG_NONBLOCK`). Архивные логи автоматически **сжимаются gzip**. Есть ротация по размеру, поддержка fork(), цветной вывод.

```c
tlog_init("logfile.log", maxlogsize, maxlogcount, buffsize, flags);
tlog(TLOG_INFO, "сообщение %d", arg);
```

**Плюсы**: процессобезопасность (уникальная особенность), gzip-сжатие, истинная асинхронность. **Минусы**: только Linux, нет конфигурационных файлов, нет категорий/правил, ограниченная документация.

### rxi/log.c и его эволюция — microlog

| Параметр | log.c | microlog |
|----------|-------|----------|
| GitHub | https://github.com/rxi/log.c | https://github.com/an-dr/microlog |
| Звёзды | **~3 300** | Новый проект |
| Лицензия | MIT | MIT |
| Размер | ~150 строк | ~2 500 строк |
| Сборка | Встраиваемый (2 файла) | CMake |
| Статус | **Не поддерживается** | Активный |

**rxi/log.c** — самый популярный минималистичный C-логгер: 6 уровней (TRACE–FATAL), printf-форматирование, вывод в stderr и файлы, поддержка callback-ов. Потокобезопасность реализуется через пользовательскую функцию блокировки. Проект не развивается (всего 7 коммитов), но код функционально завершён.

**microlog** (an-dr/microlog) — наиболее развитый форк: добавлены **8 переименовываемых уровней**, многомерная фильтрация (по выходу И по теме), отключение неиспользуемых функций на этапе компиляции, поддержка CMake-пакетов и совместимость с C++. Это **рекомендуемый выбор**, если нравится простота rxi/log.c, но нужно больше возможностей.

### spdlog — золотой стандарт (но C++)

| Параметр | Значение |
|----------|----------|
| GitHub | https://github.com/gabime/spdlog |
| Звёзды | **~28 400** |
| Лицензия | MIT |
| Язык | **C++ (C++11+)** |
| Сборка | CMake |
| Зависимости | {fmt} (в комплекте) |

**spdlog** — самая популярная библиотека логирования на GitHub с огромным сообществом. Поддерживает header-only и компилируемый режим, асинхронное логирование с пулом потоков, ротацию файлов, цветной вывод в консоль, syslog, Windows Event Log, пользовательские sinks, backtrace-буфер. Производительность — **миллионы записей/сек** в асинхронном режиме.

Критический момент: **у spdlog нет официального C API.** Для использования из C-проекта необходимо написать `extern "C"` обёртку вручную, что добавляет сложность и требует компиляции части проекта как C++. Если проект допускает C++ — это бесспорно лучший выбор. Для строго C-проектов — не подходит напрямую.

### Однофайловые и header-only библиотеки логирования

Для встраивания в проект без внешних зависимостей подходят несколько вариантов:

**zf_log** (wonder-mice/zf_log) — два файла (zf_log.h + zf_log.c), лицензия MIT. Ключевая особенность — **нулевые накладные расходы** при отключённом уровне логирования: аргументы не вычисляются. Поддерживает Android log и Apple system log (iOS, macOS). Есть hex-дамп регионов памяти и префикс ZF_LOG_LIBRARY_PREFIX для встраивания в библиотеки без конфликтов символов. Проект, к сожалению, в спящем состоянии.

**0xA1M/clog** — **одиночный заголовочный файл** с потокобезопасностью через C11 atomics, Windows Critical Sections или GCC/Clang spin-locks. Поддерживает `clog_signal_log` для безопасного логирования из обработчиков сигналов. Нет динамических аллокаций на горячем пути. Тестируется CI на GCC и Clang. Перспективный современный вариант.

**liblogax** (exoticlibraries/liblogax) — одиночный заголовочный файл, совместимый с **C90**. Уникальная особенность: поддержка **JSON-формата** и key-value вывода. Нулевые аллокации, модульная структура (ненужные функции отключаются макросами). Соответствует POSIX IEEE Std 1003.1-1988.

**lucilands/CLog** — STB-стиль (define `CLOG_IMPLEMENTATION` в одном файле). ~350 строк, пользовательские уровни, цветной вывод, защита от переполнения. C99.

### EasyLogger и embedlog — встраиваемые системы

**EasyLogger** (armink/EasyLogger) — ультралёгкий логгер для IoT и embedded: ROM < 1.6 КБ, RAM < 0.3 КБ. 6 уровней в стиле Android Logcat, фильтрация по тегам и ключевым словам, асинхронный и буферизированный режимы, плагины для Flash-хранилища и ротации файлов. Потокобезопасен. Работает на RT-Thread, UCOS, Linux, Windows, Nuttx, bare metal. Лицензия MIT.

**embedlog** (mlyszczek/embedlog) — ~800 строк, C89 (некоторые функции требуют POSIX/C99). Поддерживает stderr/stdout, файлы с ротацией, syslog, бинарное логирование для CAN-данных. Опционально реентрантный с `--enable-pthread`. **61 620 тестов пройдено**, покрытие 89.5%.

## Часть II. Альтернативы libprom — Prometheus-клиенты для C

### Текущее состояние libprom

**digitalocean/prometheus-client-c** — единственный C-клиент, перечисленный на официальной странице prometheus.io (как «неофициальный»). Репозиторий набрал **182 звезды** при всего **47 коммитах**, 26 открытых issues и 15 открытых PR. Лицензия Apache-2.0. Архитектура делится на две библиотеки: **libprom** (ядро: счётчики, gauges, гистограммы, реестр коллекторов) и **libpromhttp** (HTTP-экспозиция через libmicrohttpd). Поддерживает pthreads для потокобезопасности, использует атомарные операции с double (требуется GCC 4.8+). Есть Doxygen-документация.

Известные проблемы: краши на Debian 9, не компилировался на RHEL 7, зависимость от libmicrohttpd, Docker-ориентированный рабочий процесс разработки. **Проект фактически заброшен** — минимальная активность при значительном числе нерешённых проблем.

### jelmd/libprom — лучший выбор для чистого C

| Параметр | Значение |
|----------|----------|
| GitHub | https://github.com/jelmd/libprom |
| Звёзды | 6 (форк, не оригинал) |
| Форк от | digitalocean/prometheus-client-c |
| Лицензия | Apache-2.0 |
| Коммиты | **86** (вдвое больше оригинала) |
| Сборка | CMake + Make |
| Зависимости | libmicrohttpd, pthreads |
| Статус | **Активно развивается** |

Это **наиболее активно поддерживаемый форк** оригинальной digitalocean/prometheus-client-c, разрабатываемый Jens Elkner из Магдебургского университета. Ведётся работа по включению в **Debian** (ITP баг #1094794, январь 2025, версия 0.1.2). Используется как зависимость сборки **open5gs** (реализация ядра сети 5G), что подтверждает production-пригодность.

Улучшения по сравнению с оригиналом: исправления багов и тестов, уменьшенный объём кода с более компактными именами функций/типов, **компактный вывод** (опция подавления `# HELP` и `# TYPE` для снижения объёма), публичный `process_collector` с метриками о самом коллекторе включая время скрейпа, функции `prom_collector_data_set()`/`prom_collector_data_get()` для привязки пользовательских данных к коллекторам, `prom_counter_reset()`, Ubuntu-пакеты на pkg.cs.ovgu.de, CI через GitHub Actions.

**Рекомендация**: если проект требует чисто C Prometheus-клиент, jelmd/libprom — оптимальный выбор на сегодня. API совместим с оригиналом (`#include <libprom/prom.h>`), переход с digitalocean-версии потребует минимальных изменений.

### jupp0r/prometheus-cpp — самый зрелый вариант (C++)

| Параметр | Значение |
|----------|----------|
| GitHub | https://github.com/jupp0r/prometheus-cpp |
| Звёзды | **~1 100** |
| Лицензия | MIT |
| Язык | **C++ (современный)** |
| Сборка | CMake, Bazel |
| Зависимости | CivetWeb (pull), libcurl+zlib (push) |

Единственная библиотека в экосистеме, поддерживающая **все четыре типа метрик Prometheus**: счётчики, gauges, гистограммы и **summaries**. Встроенный HTTP-сервер на CivetWeb для pull-модели и поддержка push в Pushgateway. Доступна через vcpkg, Conan, Launchpad PPA, Bazel Central Registry. Активно поддерживается, следует Google C++ Style Guide.

**Нет C API** — использует шаблоны, `std::shared_ptr`, паттерн builder. Для использования из C необходима `extern "C"` обёртка. Если проект допускает C++-компиляцию — это наиболее надёжный и полнофункциональный выбор.

### Минималистичные и специализированные варианты

**tmlbl/prometheus.h** — **однофайловая** C-библиотека (единственный заголовочный файл). Поддерживает счётчики с лейблами, встроенный HTTP-сервер через fork(). API: `prom_register()`, `prom_get()`, `prom_metric_set()`, `prom_flush()`. Всего 6 звёзд, ограниченная документация, упрощённый сервер на fork() — но для быстрого прототипирования это самый простой путь интеграции Prometheus-метрик в C-проект.

**philbudne/libprom** — «чрезмерно тонкая» реализация Prometheus-клиента. Минимальный набор файлов, встроенный accept/dispatch для HTTP, базовые счётчики и gauges с поддержкой лейблов. Практически без документации, 2 звезды, но предельно лёгкий.

**Keenuts/prometheus-client-c** — клиент для **необычных окружений**: без libcurl, без wget, без bash, экзотические тулчейны. Поддерживает только **push в Pushgateway**. Пользователь реализует единственную функцию `void pmc_output_data(const void *bytes, size_t size)` для транспорта. Идеален для embedded-плат с нестандартным окружением.

### OpenTelemetry как путь вперёд

Проект **open-telemetry/opentelemetry-cpp** (~1 200 звёзд, Apache-2.0) — официальный CNCF C++ SDK для OpenTelemetry, поддерживающий трассировку, метрики и логи с экспортом в OTLP (gRPC, HTTP) и в Prometheus. Однако в README прямо сказано: **«Поддержка языка C не является целью проекта»**. Зависимости тяжёлые: abseil, protobuf, gRPC.

Существует экспериментальный проект **dorsal-lab/opentelemetry-c** — C-обёртка вокруг opentelemetry-cpp. Поддерживает часть API (трассировка, частично метрики). Статус: «предоставляется как есть, без гарантий и поддержки». Для production не рекомендуется, но демонстрирует принципиальную возможность использования OTel из C.

**biaks/prometheus-cpp-lite** (46 звёзд) — header-only C++ библиотека, значительно легче prometheus-cpp. Поддерживает все 4 типа метрик, запись в .prom-файлы или push на сервер. Минимальные зависимости. Не имеет C API, но обёртка будет проще, чем для полного prometheus-cpp.

### Intersec/lib-common — production-решение с оговорками

Компания **Intersec**, разрабатывающая ПО исключительно на C, создала собственный Prometheus-клиент внутри своей общей библиотеки **lib-common**. Поддерживает счётчики, gauges, гистограммы с лейблами, встроенный HTTP-сервер для скрейпинга, хелперы для таймеров гистограмм, потокобезопасность. Протестировано в production. Проблема: **невозможно использовать отдельно** — prometheus-модуль неотделим от остальной lib-common (контейнеры, аллокаторы, event loop и т.д.), а это огромная библиотека. Только Linux.

## Общие закономерности и ключевые выводы

Экосистема чистых C-библиотек для обоих направлений значительно уступает C++-аналогам по зрелости и активности поддержки. Ни одна из обнаруженных C-библиотек не рекламирует явную поддержку C17 или C23, хотя все они компилируются GCC и Clang с флагами `-std=c17`. Большинство написаны на C99/C11.

Для **логирования** ландшафт разнообразен. Если нужна максимальная функциональность на чистом C — **stumpless** (структурированное логирование, множество целей вывода, кроссплатформенность, Apache-2.0). Если приоритет — производительность и асинхронность на Linux — **pymumu/tinylog** (уникальная процессобезопасность, gzip-сжатие). Для встраиваемых систем — **EasyLogger** (ROM < 1.6 КБ). Для минималистичного встраивания — **0xA1M/clog** (потокобезопасный одиночный заголовок) или **rxi/log.c**/**microlog** (два файла, MIT). Если допустим C++ — **spdlog** вне конкуренции.

Для **Prometheus-метрик** ситуация более сложная. Экосистема C-клиентов откровенно слаба: на prometheus.io указан единственный (неофициальный) C-клиент с 47 коммитами за всю историю. Практический выбор сводится к **jelmd/libprom** (лучший форк, включается в Debian, используется в open5gs) или написанию `extern "C"` обёртки вокруг **jupp0r/prometheus-cpp**. Ни одна чистая C-библиотека не поддерживает **summaries** — только C++-решения. Однофайловая **tmlbl/prometheus.h** годится лишь для прототипов. Путь через OpenTelemetry для C пока нежизнеспособен — официальный C SDK не существует, а экспериментальные обёртки имеют статус «use at your own risk».

Стоит учитывать стратегический контекст: индустрия движется к OpenTelemetry, и Prometheus-клиенты могут в перспективе уступить место OTel SDK. Для чисто C-проектов это означает, что инвестиции в написание собственной тонкой обёртки для экспозиции метрик (формат Prometheus text exposition достаточно прост) могут оказаться более надёжным решением, чем зависимость от малоподдерживаемых библиотек.
